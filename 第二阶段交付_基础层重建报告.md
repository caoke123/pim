# 交付物 1: 新 Schema 结构说明

## 1.1 旧 Schema 问题总结

| 问题 | 旧 schema.ts | 真实 DB | 后果 |
|------|-------------|---------|------|
| 字段名 | `product_no` | `spu_code` | 所有查询默认失败 |
| SKU 外键 | `product_id` (UUID FK) | `spu_code` (varchar FK) | SKU 无法关联产品 |
| 平台存储 | 独立表 `product_platforms` | JSONB 列 `platforms_json` | 多了一层不必要的复杂度 |
| 图片结构 | `{ main, sku, detail }` 各自有 `fileName + url` | `{ main[], detail[] }` 每项有 `index + r2Url + fileName + localPath` | 图片映射错误 |
| 状态值 | `'pending'/'active'/'archived'` | `'ready'` (仅值) | 前后端状态不匹配 |
| 表覆盖 | 6 张表 (部分虚构) | 6 张真实表 (结构不同) | 迁移永远无法执行 |

| 旧混乱 | 新统一 |
|--------|--------|

| 前端 `'live'\|'pending'\|'idle'\|'error'` vs 后端无定义 | `PlatformPublishStatus`: `idle → pending → live → error` |
| 前端 `'pending'\|'running'\|'success'\|'failed'` 多处散落 | `PublishTaskStatus`: 统一 |

1. **纯数据库访问**: 不含业务逻辑、不含 API 格式化、不含权限判断

# 交付物 6: product_skus.spu_code NULL 根因分析

## 问题描述

```sql
SELECT spu_code, sku_code FROM product_skus;
-- 结果: spu_code 全部为 NULL (3/3)
```

## 根因链

```
旧 Drizzle Schema 定义
  ↓ 定义了 product_skus.product_id (UUID FK → products.id)
  ↓ 定义了 products.product_no (不存在于真实DB)
  ↓
真实 PostgreSQL 表
  ↓ product_skus 有 spu_code (varchar FK → products.spu_code)
  ↓ product_skus 没有 product_id 列
  ↓ products 有 spu_code (不是 product_no)
  ↓
旧 SyncService (services/sync.ts)
  ↓ 使用 productId (UUID) 关联 SKU
  ↓ 写入 productSkus.productId (不存在的列)
  ↓ spu_code 列从被写入
  ↓
结果:
  ↓ 3 条 product_skus 记录的 spu_code = NULL
  ↓ SKU 完全孤立, 无法与任何产品关联
```

## 直接原因

`SyncService.upsertSkus()` 方法在插入 SKU 时使用 `productId` (UUID) 作为关联键，但真实表 `product_skus` 的外键列是 `spu_code` (varchar)。旧代码从未向 `spu_code` 列写入任何值。

```typescript
// 旧代码 (错误):
skuValues: {
  productId,    // ← 这个列在真实表中不存在!
  skuCode: ...,
}

// 正确做法:
skuValues: {
  spuCode: productSpuCode,  // ← 应该用 spu_code
  skuCode: ...,
}
```

## 影响范围

- **3 条 product_skus** 完全不可用
- 前端产品详情页 SKU 标签页会显示空数据
- 平台导出无法生成 SKU 级别的行
- 分销商 API 无法返回带价格的 SKU 列表

## 修复方案 (第三阶段执行)

1. 分析 3 条 SKU 的 `sku_code` 前缀 (如 `BG-MX-0002`) 反查来源
2. 通过原始 `skus` 表或 `spus` 表定位正确的 `spu_code`
3. UPDATE 填充 `spu_code`
4. 修复 `SyncService` 使新同步的 SKU 正确写入 `spu_code`

# 交付物 7: images_json.r2Url 为空根因分析

## 问题描述

```sql
SELECT images_json->'main'->0->>'r2Url' FROM products;
-- 结果: 全部为 "" (空字符串)
```

同时: 所有 `r2_synced_at` = NULL, 所有 `r2_base_path` = NULL

## 根因

**R2 同步从未执行过。**

证据链:

| 指标 | 值 | 说明 |
|------|-----|------|
| `r2_synced_at` | 全部 NULL | R2 同步时间戳从未写入 |
| `r2_base_path` | 全部 NULL | R2 路径从未写入 |
| `r2Url` | 全部 `""` | 默认空字符串, 非 null |
| `sync_logs` (新 schema 表) | 不存在于 sorter DB | 迁移从未执行 |

## 数据来源分析

当前 `images_json` 中的数据来自**本地分拣系统**直接写入，不是 R2 同步:

```json
{
  "main": [{
    "index": 0,
    "r2Url": "",                                     ← 空
    "fileName": "主_1.png",                            ← 本地文件名
    "localPath": "D:\\图片\\素材包\\[2605-0017]...\\主_1.png"  ← Windows 本地路径
  }],
  "detail": [{ ... }]
}
```

## 结构确认

`images_json` 真实结构:
- **有** `main` 键 (数组): 产品主图
- **有** `detail` 键 (数组): 详情图
- **无** `sku` 键: SKU 图片分离存储在 `assets` 表或 `product_skus.image_url`

每张图片条目:
```typescript
{
  index: number          // 排序索引
  r2Url: string          // R2 URL (当前空)
  fileName: string       // 原始文件名
  localPath: string      // 本地绝对路径
}
```

## images_json 与 assets 表的关系

| 数据位置 | 记录数 | 内容 |
|---------|--------|------|
| `products.images_json.main[]` | 8-9 条/产品 | 产品主图 |
| `products.images_json.detail[]` | 8-9 条/产品 | 详情图 |
| `assets` 表 | 220 条 | 原始素材 (main_image/sku_image/detail_image) |

`images_json` 是 PIM 处理后的**聚合视图**，将分散素材按类型整理到产品的单一 JSON 列中。

## 修复方案 (第三阶段执行)

1. 确保 R2 连接配置正确
2. 触发 R2 同步，上传本地图片到 Cloudflare R2
3. 同步完成后填充 `r2Url` 和 `r2_base_path` / `r2_synced_at`
4. 前端图片组件需兼容: `r2Url || localPath` 的降级显示
