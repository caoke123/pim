### 架构说明

| 组件 | 定位 | 说明 |
|------|------|------|
| Docker PostgreSQL | **唯一主库 (MASTER)** | 所有业务写操作的唯一入口 |
| Supabase PostgreSQL | **云端镜像库 (REPLICA)** | 只读镜像，提供查询和灾备能力 |
| Publish Service | **同步服务** | 事件驱动，单向同步 |

### 核心原则

1. **Docker PostgreSQL 为唯一主库** — 所有业务写操作必须进入主库
2. **Supabase 为只读镜像库** — 禁止任何业务写入
3. **单向同步** — Master → Replica，禁止双向同步
4. **Schema 100% 一致** — Supabase 与本地 PostgreSQL 保持完全一致的表结构

## 2. Database Responsibility

### Master — Docker PostgreSQL

| 职责 | 说明 |
|------|------|
| 分拣系统写入 | Sorter System 写入 SPU/SKU 原始数据 |
| PIM 写入 | PIM 系统写入产品处理数据 |
| 所有业务事务 | 产品创建、编辑、删除、图册、分销等 |
| 唯一真实数据源 | 系统中唯一可信的数据源 |

**写入路径**:
```
Sorter System → Docker PostgreSQL
PIM System    → Docker PostgreSQL
```

### Replica — Supabase PostgreSQL

| 职责 | 说明 |
|------|------|
| 数据镜像 | 业务表的完整镜像 |
| 查询服务 | E-Catalog / 独立站读取 |
| 灾备恢复 | 主库损坏时的数据恢复源 |
| Catalog 读取 | 图册公开访问 |
| 独立站读取 | 未来独立站数据源 |
| BI 分析 | 数据分析（不影响主库性能） |

**禁止操作**:
- ✗ 业务写入
- ✗ 双向同步
- ✗ 直接修改数据

**读取路径**:
```
E-Catalog      → Supabase (Read Only)
Independent    → Supabase (Read Only)
BI Analytics   → Supabase (Read Only)
```

## 4. Synchronization Strategy

### 采用：Event Driven Synchronization

**禁止使用**:
- ✗ Cron Full Sync（定时全量同步）
- ✗ Database Replication（数据库复制）
- ✗ CDC（Change Data Capture）
- ✗ Debezium
- ✗ Logical Replication（逻辑复制）

### 同步流程

```
┌─────────────────────────────────────────────────────────────┐
│                    同步流程图                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌──────────────┐                                           │
│  │ Product Update│  (前端操作)                               │
│  └──────┬───────┘                                           │
│         │                                                   │
│         ▼                                                   │
│  ┌──────────────────────────┐                               │
│  │ Write Local PostgreSQL   │  (业务写入主库)                │
│  └──────────┬───────────────┘                               │
│             │                                               │
│             ▼                                               │
│  ┌──────────────────────────┐                               │
│  │ Commit Success           │  (事务提交成功)                │
│  └──────────┬───────────────┘                               │
│             │                                               │
│             ▼                                               │
│  ┌──────────────────────────┐                               │
│  │ Publish Service          │  (捕获变更事件)                │
│  │                          │                               │

│  └──────────┬───────────────┘                               │
```

### 同步要求

| 要求 | 说明 |
|------|------|
| 最终一致性 | 允许短暂延迟，最终数据一致 |
| 幂等性 | 重复同步不会产生副作用 |
| 失败重试 | 同步失败自动重试 |
| 顺序保证 | 同一记录的更新按顺序执行 |

## 5. Synchronization Failure Recovery

### sync_queue — 同步队列（未来建立）

当同步失败时，记录进入同步队列，自动重试。

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | uuid | 队列记录ID |
| `table_name` | varchar | 表名 |

| `action` | varchar | 操作类型 (INSERT/UPDATE/DELETE) |
| `payload` | jsonb | 数据负载 |

| `status` | varchar | 状态 (pending/processing/failed/completed) |
| `error_message` | text | 错误信息 |
| `created_at` | timestamptz | 创建时间 |
| `updated_at` | timestamptz | 更新时间 |
| `next_retry_at` | timestamptz | 下次重试时间 |

### 重试策略

| 重试次数 | 间隔 | 说明 |
|---------|------|------|
| 1-3 | 30秒 | 快速重试 |
| 4-6 | 5分钟 | 中期重试 |
| 7-10 | 1小时 | 长期重试 |
| >10 | 标记失败 | 人工介入 |

## 10. Architecture Decision Record (ADR)

### Decision

采用 **Master → Replica** 单向同步架构。

```
Docker PostgreSQL (MASTER) ──→ Supabase PostgreSQL (REPLICA)
```

### 拒绝方案

| 方案 | 拒绝原因 |
|------|---------|
| Dual Write | 数据一致性难以保证 |
| Dual Master | 冲突解决复杂，维护成本高 |
| Bi-Directional Sync | 循环同步风险，调试困难 |

### 理由

| 因素 | 说明 |
|------|------|
| 架构简单 | 单向同步，逻辑清晰 |
| 数据一致性高 | 单一数据源，避免冲突 |
| 维护成本低 | 团队规模小，需要简单方案 |
| 适合当前团队 | 1-2人即可维护 |

## Schema 一致性约束

### 原则

| 原则 | 说明 |
|------|------|
| 不创建 Published Schema | Supabase 使用与主库相同的 Schema |
| 不创建精简版业务表 | 业务表 100% 同步 |
| 不删除业务字段 | 保留所有字段 |
| 不修改字段类型 | 类型完全一致 |
| 不重命名字段 | 字段名完全一致 |

### 允许

| 允许 | 说明 |
|------|------|
| 新增 `synced_at` 字段 | 记录同步时间 |
| 新增同步状态辅助表 | 如 `sync_queue` |

### 目标

```
Supabase = PostgreSQL Cloud Replica
```

而不是：

```
Supabase = Published Database (已废弃)
```
