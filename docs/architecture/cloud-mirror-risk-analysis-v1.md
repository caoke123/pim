## 1. 数据同步失败

| 项目 | 内容 |
|------|------|
| **风险等级** | 🔴 高 |
| **影响** | Supabase 数据过期，E-Catalog/独立站显示旧数据 |
| **触发条件** | Publish Service 故障、网络中断、Supabase 不可用 |
| **检测方式** | 监控 `synced_at` 字段，超过阈值告警 |
| **解决方案** | 1. 自动重试机制（sync_queue）<br>2. 告警通知（企业微信/邮件）<br>3. 人工介入修复<br>4. 失败记录持久化，服务恢复后自动补偿 |
| **缓解措施** | 1. 设置同步延迟阈值（如 5 分钟）<br>2. E-Catalog 页面显示数据更新时间<br>3. 定期全量校验 |

## 2. Supabase 不可用

| 项目 | 内容 |
|------|------|
| **风险等级** | 🟡 中 |
| **影响** | E-Catalog/独立站无法访问，但 PIM 系统正常运行 |
| **触发条件** | Supabase 服务中断、网络问题、账户欠费 |
| **检测方式** | 健康检查端点，定期 ping Supabase |
| **解决方案** | 1. E-Catalog 启用 Cloudflare 缓存（降级模式）<br>2. 缓存过期前自动刷新<br>3. Supabase 恢复后自动同步<br>4. 备用数据源（可选） |
| **缓解措施** | 1. Cloudflare Workers 缓存产品数据<br>2. 设置合理的缓存 TTL（如 1 小时）<br>3. 缓存降级页面 |

---

## 3. 主库损坏

| 项目 | 内容 |
|------|------|
| **风险等级** | 🔴 高 |
| **影响** | PIM 系统完全不可用，业务中断 |
| **触发条件** | 硬盘损坏、误操作删除、数据库崩溃 |
| **检测方式** | 数据库健康检查、备份监控 |
| **解决方案** | 1. 从 Supabase 恢复数据（灾备流程）<br>2. 新服务器部署 PostgreSQL<br>3. 导出 Supabase 数据导入新库<br>4. 验证数据完整性后恢复服务 |
| **缓解措施** | 1. 定期本地备份（每日）<br>2. Supabase 作为云端灾备<br>3. 准备灾备脚本（一键恢复）<br>4. RTO < 8 小时 |

## 4. Tunnel 中断

| 项目 | 内容 |
|------|------|
| **风险等级** | 🟡 中 |
| **影响** | 远程办公员工无法访问 PIM 系统 |
| **触发条件** | Cloudflare Tunnel 故障、网络问题、配置错误 |
| **检测方式** | Tunnel 健康检查、员工反馈 |
| **解决方案** | 1. 重启 Cloudflare Tunnel<br>2. 检查网络配置<br>3. 临时方案：VPN 访问<br>4. 本地办公不受影响 |
| **缓解措施** | 1. 配置 Tunnel 自动重启<br>2. 备用 VPN 方案<br>3. 员工本地缓存工作文件 |

## 5. 数据不一致

| 项目 | 内容 |
|------|------|
| **风险等级** | 🟠 中高 |
| **影响** | Supabase 与主库数据不一致，下游系统显示错误数据 |
| **触发条件** | 同步逻辑 Bug、并发写入、网络分区、部分同步失败 |
| **检测方式** | 1. 定期数据校验（checksum）<br>2. 记录数对比<br>3. 关键字段抽样对比 |
| **解决方案** | 1. 自动检测不一致<br>2. 触发全量同步修复<br>3. 定位不一致原因<br>4. 修复同步逻辑 Bug |
| **缓解措施** | 1. 幂等性设计（Upsert）<br>2. 同步队列保证顺序<br>3. 定期全量校验（每日）<br>4. E-Catalog 显示数据更新时间 |
